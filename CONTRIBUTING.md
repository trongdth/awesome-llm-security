# Contributing to Awesome LLM Security

Thank you for your interest in contributing to this awesome list!

Please follow these guidelines to make sure your contribution can be merged quickly and seamlessly.

## Curation Policy

To maintain high quality, not every paper on Arxiv or every new tool will be accepted. Our categorization focuses heavily on real-world applicability:

1. **Core Resources**: Foundational documents and industry-standard tools (e.g., OWASP, Garak).
2. **Emerging Tech**: The latest research (e.g., 2025-2026 developments like MCP or Agentic AI).
3. **Community Contributions**: Small tools, scripts, or personal blogs. These should clearly state their security use-case.

## Guidelines for Pull Requests

- First, **ensure the resource is truly "Awesome"** (highly useful, high quality, well-maintained).
- Search previous suggestions or PRs before making a new one to avoid duplicates.
- Make an individual pull request for each suggestion.
- The pull request and commit should have a descriptive title and message.
- Use the following format for lists: `- [Title/Name](Link) - Short, descriptive summary starting with a capital letter and ending with a period.`
- Priority is given to updated and recent resources (2025-2026).
- Explain **why** the resource is "Awesome" in your PR description.
- Ensure the link is functional and directly leads to the resource.

## Labels

Please use the provided issue/PR labels correctly to help reviewers triage your contribution (e.g., `paper`, `tool`, `guideline`, `benchmark`, `agentic-ai`, `llm-security`).

We encourage peer review! Feel free to review other's PRs to ease the maintainer's load and foster a learning community.
